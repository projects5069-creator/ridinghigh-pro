## 2. ⚡ TL;DR — 60-Second Overview

**RidingHigh Pro** is an automated short-selling research system. It scans
the US equity market every minute, identifies stocks pumping unusually hard,
scores each one 0-100, simulates a $1,000 short trade with TP=−10% / SL=+10%,
and tracks the outcome over 5 trading days.

**No real money is traded.** The system runs in Phase 1: data accumulation
and edge validation. After ~100 clean Score v2 records and verified
TP10 hit-rate stability, the system may transition to live execution.

**Status as of 2026-05-02:**
- 156 records collected (104 v1 deprecated, 50 v2 valid, 2 May v2)
- TP10 hit rate (v2): 75.0%
- Score predictive power: r=0.045 (essentially zero — see ADR-002)
- 18 health checks running 3×/day with email alerts
- Alpaca paper trading active, all systems green

## 3. 🧠 Mental Model

> **"RidingHigh Pro is like a fishing fleet that doesn't fish."**

Imagine a fleet of boats casting nets every minute across a vast ocean (the
US stock market). The boats don't actually catch fish (no real trades).
Instead, every fish that swims into a net is photographed, measured,
weighed, scored, and released. Months later, the records reveal which
patterns of fish (which signal combinations) reliably predict which fish
will be edible (profitable shorts).

**Only when patterns prove statistically reliable will the boats start
catching real fish.**

This mental model implies:

- **Speed of catch matters less than completeness of records.** A missed
  scan is acceptable; a corrupted record is not.
- **The catch is the data, not the money.** Every scan, every score,
  every D1-D5 OHLC is the asset.
- **Patience is structural.** Phase 1 lasts months by design.
- **Score is a hypothesis, not a verdict.** Until correlation with wins is
  proven, Score is a guess.

When you ask Claude "should we add X to the scanner?" — the answer
depends on whether X improves catch quality (good) or just adds noise
to the records (bad).

## 6. 🎯 System Purpose & Thesis

### The thesis

Stocks that experience extreme single-day price increases ("pumps") on
elevated volume tend to mean-revert in the days that follow. The thesis
is empirical, not philosophical: certain combinations of pump signals
(low liquidity, high relative volume, intraday range exceeding ATR)
should produce statistically biased short opportunities.

### What the system tries to prove

**H₀ (null hypothesis):** Score has no predictive power above random
selection (TP10 hit rate ≈ 50%).

**H₁ (working hypothesis):** A composite Score that weights MxV, RunUp,
ATRX, RSI, VWAP_dist, ScanChange, and REL_VOL identifies short candidates
with TP10 hit rate ≥ 65% over a 5-day window.

### Current evidence (2026-05-02)

- **TP10 hit rate (v2):** 75.0% — **above** the H₁ threshold ✅
- **Score correlation with hits:** r=0.045 — **near zero** ❌

The dissonance is the central research finding: hit rate is high, but
Score doesn't drive it. This suggests the **filter** (which stocks
even appear) is doing the work, while Score may be measuring pump
intensity rather than reversion probability. See **ADR-002**.

### Architecture principles

1. **No formula duplication.** All metrics flow through `formulas.py`.
   This was enforced after early bugs where the same formula existed in
   `auto_scanner` and `dashboard` and diverged silently.

2. **Provider abstraction.** Calling code does not know whether prices
   come from Alpaca or yfinance. The active provider is selected via
   `DATA_PROVIDER` env var.

3. **Config centralization.** Every threshold, weight, and tunable lives
   in `config.py`. The `code_auditor.py` flags hardcoded values.

4. **Read-once, write-once for sheets.** No concurrent writers. Monthly
   rotation creates fresh sheets to avoid quota and integrity issues.

5. **Idempotent maintenance.** Monthly rotation, OAuth warming, and
   backups can run multiple times safely.

| **Alpaca** | "best effort" (paper) | No prices for verification | Auto-fallback to yfinance via `data_provider.py` |
| **Yahoo Finance** | none | No fundamentals for new tickers | Stale `data/market_cap_cache.json` used |
| **Google Sheets** | 99.9% per Google | No data persistence | Quota guard in `gsheets_sync.py`; CSV backup 3×/day |
| **GitHub Actions** | varies | No automation | Manual `python3 auto_scanner.py` from local |
| **cron-job.org** | none (free) | No minute-level triggers | `auto_scan.yml` also has internal cron `* * * * *` as backup |
| **Gmail SMTP** | high | No email alerts | Health audit still runs and writes to sheet |
| **Streamlit Cloud** | best-effort free | No dashboard | Sheets accessible directly via Google UI |

### Single points of failure (no fallback)

1. **FINVIZ** — there is no automatic fallback for the screener.
2. **GitHub** — code, secrets, and automation all live there.
3. **Google account** — Sheets, Drive, OAuth all tied to one account.

These are accepted risks for a single-operator research system.

## 13. 📊 Data Sources

| Source | Role | Endpoint | Update freq | Cost |
|--------|------|----------|-------------|------|
| **FINVIZ** | Pre-market filter; raw signal source | Web scrape via `finvizfinance` | Real-time | Free |
| **Alpaca** | Verification prices, D1-D5 OHLC | SIP historical, IEX live (paper account) | Real-time | Free (paper) |
| **yfinance** | Fundamentals (Float, Sector, Industry, etc.) + fallback prices | Yahoo public API | Real-time | Free, rate-limited |

### Why three sources

- **FINVIZ** is the only source that provides cheap, real-time market-wide
  screening with the filters we need. It is the funnel.
- **Alpaca** is the price-of-record for the system. Its data feeds
  simulation today and live trades tomorrow (Phase 4).
- **yfinance** provides what Alpaca does not: floatShares, sharesOutstanding,
  sector, industry, P/E, beta. It also serves as failover.

### Critical data quirks

- ⚠️ **FINVIZ Change% comes as a decimal** (e.g., `0.3726`). Must be
  multiplied by 100 to get a percentage. Hardcoded handling in
  `auto_scanner.analyze_ticker()`.
- ⚠️ **yfinance returns NaN in pre-market** for some tickers — handled
  with retries in providers.
- ⚠️ **Alpaca SIP feed has 15-minute delay on free tier for live quotes;
  historical bars are fine.** This is acceptable because the scanner uses
  FINVIZ for live prices and Alpaca only for verification + historical.
- ⚠️ **REL_VOL outliers** can be in the thousands (yfinance bug). Capped
  at 100.0 in `formulas.calculate_rel_vol()` via `REL_VOL_CAP`.

## 16. 🧬 Data Lineage Examples

### Example 1: How `TP10_Hit` is computed

```
SOURCES:
  ScanPrice  → FINVIZ snapshot (last trade)
                ↓
                logged in timeline_live by auto_scanner.run_scan()
                ↓
                copied to post_analysis on EOD
  D1-D5 OHLC → Alpaca historical bars (provider abstraction)
                ↓
                fetched by post_analysis_collector.fetch_ohlc_for_days()
                ↓ (or)
                fetched by backfill_ohlc.fetch_ohlc() if missed

COMPUTATION:
  formulas.calculate_max_drop(min_low, scan_price)
    where min_low = min(D1.low, D2.low, D3.low, D4.low, D5.low)
  TP10_Hit = (max_drop / scan_price) ≤ -0.10
            = max_drop_pct ≤ -10%

WRITTEN BY:
  post_analysis_collector.py · run() → save_post_analysis_to_sheets()

READ BY:
  dashboard.py · daily_summary_page() and timeline_archive_page()
  Aggregations: hit rate, win rate, conditional win rate by Score tier

VALIDATED BY:
  health_audit.check_05 (post_analysis completeness)
  health_audit.check_07 (score range)
  health_audit.check_09 (duplicates)
```

### Example 2: How `Score` flows from raw FINVIZ to dashboard

```
1. FINVIZ row arrives in auto_scanner.fetch_finviz()
2. analyze_ticker() called per-row
3. Raw fields parsed: parse_market_cap, parse_volume (utils.py)
4. Each metric calculated via formulas.py (no inline math)
5. calculate_score(metrics) returns 0-100
6. Row written to timeline_live with all metrics + Score
7. Best-of-day deduplication writes to daily_snapshots
8. EOD collector writes selected rows to post_analysis
9. Dashboard reads cached_post_analysis() and renders
```

### Why lineage matters

When debugging an anomaly like *"Why did ticker X get Score 100?"*, the
lineage tells you exactly which file to read first (formulas.py:380),
which input was responsible, and which sheet to inspect for raw values.

## 18. 🧮 Score v2 — Weights, Caps, Formula

### Weights (sum must = 100)

```python
SCORE_WEIGHTS_V2 = {
    "MxV":        25,   # illiquidity signal
    "RunUp":      25,   # pump intensity
    "ATRX":       20,   # volatility expansion
    "RSI":        10,   # overbought (with bell curve, peak 50-70)
    "VWAP":       10,   # price above VWAP
    "ScanChange": 5,    # change since prev close
    "REL_VOL":    5,    # unusual volume
}
```

### Caps (per-metric thresholds for max contribution)

```python
SCORE_CAPS_V2 = {
    "MxV":        200,  # abs value (scores based on |mxv|/200)
    "RunUp":      30,   # percentage
    "ATRX":       5,    # ratio (range/ATR)
    "RSI_HIGH":   70,   # bell curve center high
    "RSI_LOW":    50,   # bell curve center low (alias)
    "VWAP":       8,    # percentage
    "ScanChange": 60,   # percentage
    "REL_VOL":    15,   # ratio (for score contribution; raw cap is 100)
}
```

### Score tiers (display)

| Range | Tier | Color | Action |
|-------|------|-------|--------|
| 85+ | Critical | 🔴 Red | Strongest signal — flagged |
| 60-84 | High | 🟠 Orange | Above display threshold |
| 40-59 | Medium | 🟡 Yellow | Watch only |
| 0-39 | Low | ⚪ Gray | Filtered out |

### Action thresholds

| Constant | Value | Meaning |
|----------|-------|---------|
| `MIN_SCORE_DISPLAY` | 60 | Minimum to show in dashboards |
| `TRADE_ENTRY_MIN_SCORE` | 70 | Minimum to enter portfolio simulation |
| `CRITICAL_SCORE` | 85 | Critical tier |
| `HIGH_SCORE` | 60 | High tier |
| `MEDIUM_SCORE` | 40 | Medium tier |

### Score v2 formula (high level)

```
For each metric m in [MxV, RunUp, ATRX, RSI, VWAP, ScanChange, REL_VOL]:
    contribution_m = weight_m × clamp_to_cap(raw_m, cap_m)
score = sum(contributions) capped at 100
```

The exact implementation lives in `formulas.calculate_score()` and is the
**canonical authoritative version**.  Any other implementation anywhere
in the codebase is a bug.

## 20. 💼 Trade Simulation

### The simulated short trade

For each stock with Score ≥ `TRADE_ENTRY_MIN_SCORE` (70):

```
Entry:    BORROW & SELL at entry price (short)
              ↓
              wait up to 5 trading days (D1-D5)
              ↓
Watch every minute (live_trades sheet) and EOD (portfolio sheet):
  IF intraday_low ≤ entry × 0.90  →  TP10 HIT  (WIN, +$100 on $1000)
  IF intraday_high ≥ entry × 1.10 →  SL HIT    (LOSS, -$100 on $1000)
  IF day 5 ends with neither hit  →  TIMEOUT   (close at D5 close, +/- accordingly)

Position size: fixed $1,000 per trade
SL ALWAYS overrides TP if both hit on the same bar (conservative)
```

### Two parallel tables

| Table | Entry price | Reasoning |
|-------|-------------|-----------|
| **Table A** | `ScanPrice` (EOD of scan day) | Theoretical: enter at signal time |
| **Table B** | `D1_Open` (next-day open) | Realistic: gap risk included |

Historical performance gap: Table A ~80% TP10 vs Table B ~60% — see ADR-003.
The difference is itself useful data (shows magnitude of overnight gap risk).

### Status colors (in dashboard)

| Color | Meaning |
|-------|---------|
| 🟢 Green | TP10 hit — WIN |
| 🔴 Red | SL hit — LOSS |
| 🟡 Yellow | Pending / Open (no exit yet) |
| ⚪ Gray | Closed at timeout (D5) |

### `portfolio_live` real-time tracking

During market hours, every minute:
- Fetch current price for every Open position
- Update `RunningHigh`, `RunningLow`
- If TP/SL levels crossed → close position with appropriate status
- Calculate intermediate `PnL` for live display

## 21. ⚙️ 7 Workflows

All workflows live in `.github/workflows/`. Times shown are Peru (UTC-5).

### `auto_scan.yml` — the main scanner
- **Trigger:** `cron: '* * * * *'` (every minute, all the time)
- **Filter:** Logic inside `auto_scanner.py` checks `is_market_hours()`
  and exits cleanly outside market window. cron-job.org also sends dispatch
  triggers as redundancy.
- **Steps:** install deps → run `python auto_scanner.py`
- **Env:** `GOOGLE_CREDENTIALS_JSON`, `GOOGLE_OAUTH_TOKEN_JSON`,
  `ALPACA_API_KEY_ID`, `ALPACA_SECRET_KEY`, `DATA_PROVIDER=alpaca`,
  `ALPACA_PAPER=true`
- **Timeout:** 8 minutes
- **Typical runtime:** 30-90 seconds per scan

### `post_analysis.yml` — daily research collector
- **Trigger:** `cron: '5 21 * * 1-5'` (16:05 Peru, Mon-Fri)
- **Steps (sequential):**
  1. `python auto_scanner.py --eod` — EOD snapshot
  2. `python post_analysis_collector.py` — fetch 5-day OHLC
  3. `python enrich_post_analysis.py` — intraday enrichment
  4. `python backfill_ohlc.py` — fill gaps from prior days
- **Env:** Alpaca + Google secrets
- **Timeout:** 15 minutes

### `health_audit.yml` — system health
- **Triggers (3×/day, every day including weekends):**
  - `0 11 * * *` (06:00 Peru, pre-market)
  - `0 17 * * *` (12:00 Peru, mid-day)
  - `0 3 * * *` (22:00 Peru — actually 03:00 UTC next day)
- **Steps:** install deps → `python3 health_audit.py`
- **Env:** Google secrets, Alpaca, Gmail SMTP, REPORT_TO
- **Continues on error:** Yes — failure to send email does not abort report

### `backup.yml` — daily CSV backups
- **Triggers (3 cron times to span the day):**
  - `7 13-21 * * 1-5` (every hour from 08:07 to 16:07 Peru)
  - `7 20 * * 1-5` (15:07 Peru, secondary)
  - `30 21 * * 1-5` (16:30 Peru, post-EOD)
- **Steps:** install deps → `python backup_manager.py`
- **Output:** `backups/post_analysis_*.csv` uploaded as GitHub artifact
  (90-day retention, no Drive quota)

### `monthly_rotation.yml` — month-1 active switch
- **Trigger:** `cron: '1 5 1 * *'` (00:01 Peru, 1st of month)
- **Steps:** `python monthly_rotation.py` → commit `sheets_config.json`
- **Permission:** `contents: write` (pushes to repo)

### `prepare_next_month.yml` — pre-create next-next month
- **Trigger:** `cron: '5 5 1 * *'` (00:05 Peru, 1st of month, after rotation)
- **Steps:** `python3 prepare_next_month.py` → commit config
- **Why offset 4 minutes after rotation:** rotation must complete first

### `warm_oauth_token.yml` — OAuth token preservation
- **Trigger:** `cron: '0 12 */3 * *'` (07:00 Peru, every 3 days)
- **Why:** Google revokes refresh tokens after 7 days inactivity for apps
  in 'Testing' OAuth consent. Periodic refresh prevents this.
- **Failure mode:** sends email with manual renewal instructions if token
  cannot be refreshed.

## 26. 💾 Recovery & Disaster Recovery (DR)

### Recovery Point Objective (RPO)
The maximum acceptable data loss is **1 trading day**.

### Recovery Time Objective (RTO)
The maximum acceptable downtime is **48 hours** (one weekend).

### Recovery assets (where everything lives)

| Asset | Primary | Backup | Tertiary |
|-------|---------|--------|----------|
| **Code** | GitHub repo (canonical) | Local clone on Mac | None needed |
| **post_analysis data** | Google Sheets | CSV in `backups/` | GitHub artifacts (90-day) |
| **Other sheets** | Google Sheets | Manual export only | None |
| **Secrets** | GitHub Secrets | (TBD: 1Password recommended) | None |
| **OAuth token** | Auto-refreshed every 3 days | Manual re-grant via `get_oauth_token.py` | None |

### Disaster scenarios & playbooks

#### Scenario 1: Local Mac dies
- **Impact:** None to running system. Local-only tooling unavailable.
- **Recovery:**
  1. Get a new Mac
  2. `git clone https://github.com/projects5069-creator/ridinghigh-pro.git ~/RidingHighPro`
  3. Copy `google_credentials.json` from secure backup
  4. `pip install -r requirements.txt`
  5. Test: `python3 morning_health_check.py`
- **RTO:** ~2 hours

#### Scenario 2: Sheet accidentally deleted
- **Impact:** Loss of in-progress data; backups partial.
- **Recovery:**
  1. Check Google Drive trash (30-day recovery window)
  2. If lost: restore from CSV backup in `backups/` or GH artifact
  3. Use `migrate_*` scripts to repopulate if available
- **RTO:** 4-8 hours

#### Scenario 3: GitHub Actions disabled / repo wiped
- **Impact:** No automation. Existing data preserved in Sheets.
- **Recovery:**
  1. Re-create repo from local clone: `git push <new-url>`
  2. Re-add all secrets
  3. Re-enable Actions
  4. Trigger workflows manually to verify
- **RTO:** 4 hours

#### Scenario 4: Google account suspended
- **Impact:** Catastrophic. Sheets, Drive, OAuth all gone.
- **Recovery:**
  1. Recover Google account or create new
  2. Set up new project, service account
  3. Re-create sheet structure (`prepare_next_month.py` for one month)
  4. Update `sheets_config.json` and `google_credentials.json`
  5. Historical data in CSV backups (≤90 days) and Drive backups (manual)
- **RTO:** 24-48 hours
- **Data loss:** Anything not backed up to CSV. Up to weeks of in-Sheets data
  not captured by `backup_manager.py`.

#### Scenario 5: Alpaca account suspended
- **Impact:** Provider abstraction kicks in — yfinance fallback engages.
- **Recovery:** Sign up for new Alpaca paper account; update env vars.
- **RTO:** 1 hour
- **Note:** No data loss; system runs on yfinance during outage.

**Cause:** `warm_oauth_token.py` could not refresh.

**Steps:**
1. SSH/local to Mac
2. `cd ~/RidingHighPro && python3 get_oauth_token.py`
3. Follow browser flow → grant access → token saved to local file
4. Copy token JSON content
5. Update GitHub Secret `GOOGLE_OAUTH_TOKEN_JSON`
6. Trigger `warm_oauth_token.yml` manually to verify
7. Wait for next health audit — should clear

#### Incident: No emails arriving

**Diagnosis:**
1. Check spam folder
2. Check Gmail "All Mail" — search for "RidingHigh"
3. Check `health_audit.yml` in GitHub Actions tab — recent runs?
4. If runs are succeeding but no email: secret `GMAIL_APP_PASS` may be
   revoked. Generate new app password at Google Account → Security.

#### Incident: Sheets quota exceeded (rare)

**Symptoms:** `health_audit.check_04/05/07` failures showing "API quota
exceeded" type errors.

**Steps:**
1. Wait 100 seconds (the API limit window)
2. Retry the failing operation manually
3. If recurring, reduce dashboard refresh frequency or increase cache TTL

#### Incident: post_analysis missing today's row

**Causes:**
- Scanner didn't run today (check_04 should have caught it)
- Collector failed (check_05 should have caught it)
- Today's stocks all had Score < 60 (legitimate)

**Steps:**
1. Check `daily_summary` for today — were there any stocks?
2. If yes but post_analysis empty: run `python post_analysis_collector.py`
3. Then `python backfill_ohlc.py` to fill any missing OHLC

#### Incident: Dashboard shows stale data

**Causes:**
- Cache hasn't expired (5 min default)
- Streamlit Cloud cold-started

**Steps:**
1. Click "🗑️ Clear Local Cache" button in sidebar
2. Click "🔄 Refresh data"
3. If still stale: check sheet directly in Google Drive

## 30. 📜 Architecture Decision Records (ADRs)

ADRs document **why** key technical decisions were made. They never get
deleted; superseded ones are marked but kept for context.

### ADR-001: Score v1 → v2 transition (2026-04-11, commit f3d96ca)

- **Status:** Accepted
- **Context:** Score v1 (the original) had 7 metrics with weights:
  MxV=30, RunUp=20, REL_VOL=20, RSI=10, ATRX=10, Gap=5, VWAP=5.
  Empirical correlation analysis suggested some weights were misaligned.
- **Decision:** Replace with Score v2:
  MxV=25, RunUp=25, ATRX=20, RSI=10, VWAP=10, ScanChange=5, REL_VOL=5.
  Drop Gap entirely. Add ScanChange. Reweight ATRX up, REL_VOL down.
- **Consequences:**
  - 104 historical records (v1) became inconsistent with new formula
  - Per ADR-005, rather than recompute, those records were tagged
    `score_version='v1'` for filtering.
  - Score predictive power did not improve (r still ~0.05).
- **Status notes:** v1 records remain in post_analysis but are filtered
  out of all primary analyses by `score_version='v2'` filter.

### ADR-002: Why Score-TP10 correlation is near zero, even with v2

- **Status:** Active investigation
- **Context:** TP10 hit rate is 75-80% (excellent), but
  correlation between Score and TP10 is r≈0.05 (essentially zero).
- **Working theory:** The **filter** (`MIN_SCORE_DISPLAY=60` and
  fundamental gates like min volume, min market cap) is doing the
  predictive work. Once a stock passes the filter, its specific Score
  inside the 60-100 range carries no additional signal.
- **Implication:** Score may need redesign in v3 (or eliminated as
  a binary flag rather than a continuous predictor).
- **Action:** Phase 1 must continue accumulating v2 data to confirm.
  Score v3 design deferred.

### ADR-003: Two trade tables (A vs B)

- **Status:** Accepted
- **Context:** Need to model both theoretical (enter at scan) and
  realistic (enter at next-day open) trade outcomes.
- **Decision:** Maintain two parallel simulations.
- **Consequences:** Table A's outperformance over Table B (~80% vs ~60%)
  quantifies overnight gap risk — useful for sizing real Phase 4 trades.

### ADR-004: Tag instead of recompute v1 records (2026-05-02, commit bf2892b)

- **Status:** Accepted (Issue #N10 closed)
- **Context:** 104 of 156 records (67%) had Score v1. Fully recomputing
  v2 over 5 sheets would take ~6 hours and risk inconsistencies.
- **Decision:** Add `score_version` column to `post_analysis` and
  `daily_snapshots`. Tag each row v1 or v2. Filter analyses to v2.
- **Consequences:**
  - Raw metrics (MxV, ATRX, RunUp) remain unchanged → still usable for
    primary analyses regardless of tag.
  - 30-minute fix vs 6-hour fix.
  - Idempotent script (`mark_score_version.py`) safe to re-run.
- **Considered alternatives:** Full recompute (rejected: too risky given
  r=0.05 marginal value of Score itself).

### ADR-005: Provider abstraction (data_provider.py, 2026-04-25, Issue #9)

- **Status:** Accepted (Phase 2 of Issue #9)
- **Context:** yfinance was the only data source. Risks: rate limits,
  silent NaN returns, no SLA. For Phase 4 (real money), need an actual
  broker connection anyway.
- **Decision:** Build `data_provider.py` abstract base class with two
  implementations: `AlpacaDataProvider` (primary, paper trading) and
  `YFinanceDataProvider` (fallback + fundamentals).
- **Consequences:**
  - Calling code (collector, scanner, dashboard) is provider-agnostic.
  - Switching provider is one env var change: `DATA_PROVIDER=alpaca`.
  - Testable via `validate_providers.py` A/B comparison.
  - Phase 4 transition will reuse same code, just unset `ALPACA_PAPER`.

### ADR-006: 9 sheets per month, not 1 sheet with 9 tabs

- **Status:** Accepted
- **Context:** Original design used one sheet with multiple tabs.
  Hit Sheets API quota (per-sheet per-100-seconds limits).
- **Decision:** Each "tab" became its own sheet file. Total of 9 per
  month. Folder per month in Drive.
- **Consequences:**
  - Quota distributed across files (each sheet has its own quota bucket).
  - More complex config (one ID per sheet).
  - Easier rotation (pre-create files for next month idempotently).

### ADR-007: TP=10%, SL=10% (changed from SL=7%, Issue #1)

- **Status:** Accepted
- **Context:** Initial design had asymmetric TP=-10%, SL=+7%. Across
  early data, +7% SL was triggered too often by normal noise, causing
  premature losses on positions that would later have hit TP10.
- **Decision:** Symmetric ±10%.
- **Consequences:**
  - Win rate improved (fewer premature SL exits).
  - Position size kept at $1000 (-/+$100 outcomes).
  - SL still overrides TP if both hit same bar (conservative).

### ADR-008: Single-file dashboard.py (not multi-file Streamlit pages)

- **Status:** Accepted (with caveat — see roadmap)
- **Context:** Streamlit supports multi-page apps via `pages/` folder.
  Considered splitting dashboard.
- **Decision:** Keep single-file (currently 1,898+ lines).
- **Reasoning:** State sharing simpler in single-file; no pages/ overhead.
  Three pages (Home / Daily Summary / Timeline Archive) implemented as
  radio button + conditional rendering.
- **Future:** When System Overview page is added (per #N26), may split.

## A7. 🧠 Agent Decision Logic (Phase 1, M3)

The trader's brain. Takes a raw signal + account state, returns a
Decision with full reasoning.

**Status:** Implemented 2026-05-03 (Milestone 3)

### Module structure

| File | Purpose | Lines |
|------|---------|-------|

### Decision flow (10 filters in order)

1. SCORE_TOO_LOW: score < AGENT_MIN_SCORE (60)
2. MXV_TOO_HIGH: mxv > AGENT_MXV_MAX (-100)
3. RUNUP_TOO_LOW: run_up < AGENT_RUNUP_MIN (30%)
4. VOLUME_TOO_LOW: volume < AGENT_VOLUME_MIN (100K)
5. MARKET_CAP out of range: [$5M, $2B]
6. QUALITY_TOO_LOW: quality_score < 0.5 (>=3 flags)
7. EXISTING_POSITION: already short this ticker
8. COLD_START_LIMIT: concurrent (5) or daily (10) cap reached
9. INSUFFICIENT_BUYING_POWER: < $1000

If all 9 pass → ENTER. Otherwise → SKIP with specific skip_reason.

### Critical: stateless design

The Trader does NOT cache state between calls. Account state
(open positions, cold_start counters, buying_power) must be passed
as `account_state` parameter. This makes:
- Testing trivial (no fixture state)
- Concurrency safe (no shared state)
- M5 transition simple (state moves to Alpaca, no refactor)

## A9. ⚡ Agent Execution Layer (Phase 1, M5)

The order execution layer. Translates ENTER decisions into actual
broker calls (Alpaca paper) and tracks position lifecycle.

**Status:** Implemented 2026-05-03 (Milestone 5)

### Module structure

| File | Purpose | Lines |
|------|---------|-------|
| `agent/execution/alpaca_broker.py` | Alpaca SDK wrapper, DRY_RUN simulation, safety guards | 270 |
| `agent/execution/order_manager.py` | Bracket order submission, retry, paper_portfolio writes | 265 |
| `agent/execution/position_manager.py` | Open position monitoring, TP/SL detection, EOD close | 250 |
| `agent/execution/reconciler.py` | Sheet ↔ Alpaca drift detection and correction | 218 |

### Critical safety architecture

Three independent guards prevent accidental live trading:

1. **Broker-level**: `assert "paper" in base_url` in `AlpacaBroker.__init__`
2. **Key-level**: API key must start with "PK" (paper convention)
3. **State-level**: `AGENT_LIVE_PAPER` flag blocks `submit_bracket_order`
   when False (current state). Only M10 enables this.

State machine:
- `AGENT_DRY_RUN=True` → simulated orders, no API calls (current)
- `AGENT_DRY_RUN=False + AGENT_LIVE_PAPER=False` → blocked (RuntimeError)
- `AGENT_DRY_RUN=False + AGENT_LIVE_PAPER=True` → executes (M10 enables)

### Order flow (ENTER decision → completed position)

1. Trader returns Decision(action="ENTER")
2. OrderManager.execute(decision):
   - Pre-flight: market open, broker available
   - Submit bracket order (entry limit + TP + SL as OCO)
   - Poll status up to 60s, cancel on timeout
   - Update Decision: order_id, order_status, execution_price
   - Write to paper_portfolio Sheet (Status="OPEN" or "DRY_RUN_OPEN")
3. PositionManager.monitor_all() (per-minute loop):
   - For each open position: check TP/SL fills
   - Update CurrentPrice, UnrealizedPnL/Pct
   - On TP filled: close position with ExitReason="TP_HIT"
   - On SL filled: close position with ExitReason="SL_HIT"
4. PositionManager.eod_close_all() (14:55 Peru):
   - Close all open positions before market close
   - Cancel dangling TP/SL orders

### Reconciler drift handling

Runs daily after market close. 4 scenarios:

| Sheet Status | Alpaca Reality | Action |
|--------------|----------------|--------|
| OPEN | Position exists | OK (no action) |
| OPEN | No position | PHANTOM_OPEN: mark CLOSED + alert |
| (none) | Position exists | ORPHAN_POSITION: alert only (don't auto-close) |
| DRY_RUN_OPEN | (any) | Skipped (no Alpaca counterpart) |

### Tradability upgrade (M5)

`agent/perception/tradability.py` evolved from M3 mock to broker-aware:
- No broker passed → mock fallback (M3 behavior preserved)
- AGENT_DRY_RUN=True → mock fallback regardless
- Broker provided + LIVE → real `broker.is_shortable()` with 1-hour cache
- On API failure → graceful fallback to mock

### Retry strategy (Spec §8.4)

- 3 retries with exponential backoff: 2s, 4s, 8s
- No retry on rejection (logical error, not transient)
- After all retries exhausted → mark Decision REJECTED, alert

### DRY_RUN markers in paper_portfolio

DRY_RUN positions are clearly distinguishable:
- `Status`: "DRY_RUN_OPEN" (vs "OPEN" for LIVE)
- `EntryOrderID`: "SIM-{uuid}" prefix
- `TPOrderID`/`SLOrderID`: "SIM-{uuid}-tp", "SIM-{uuid}-sl"

This allows M7 Score Analytics to filter LIVE vs DRY_RUN trades.

## A10. 📝 Agent Postmortem Engine (Phase 1, M6)

Generates postmortem analysis for closed positions. Foundation for
M7 Score Analytics correlations.

**Status:** Implemented 2026-05-03 (Milestone 6)

### Module structure

| File | Purpose | Lines |
|------|---------|-------|
| `agent/analytics/postmortem_engine.py` | Per-position postmortem generator | 346 |

### Trigger

Per-position: hooked into `position_manager._close_position()`. After
position is marked CLOSED in paper_portfolio, postmortem generates
within ~1 minute (acceptance criteria from spec).

### postmortems Sheet schema (17 columns)

PostmortemID, PositionID, Ticker,
EntryDate, EntryPrice, ScoreAtEntry, MetricsAtEntry,
ExitDate, ExitPrice, PnLPct, ExitReason,
DurationHours, MaxFavorable, MaxAdverse, AutoLessons,
GeneratedAt, ScoreVersion

### Computed metrics

| Metric | Formula | Source |
|--------|---------|--------|
| DurationHours | (exit_dt - entry_dt) in hours | paper_portfolio dates |
| MaxFavorable (MFE) | min(Low) during hold (short: lower=better) | data_provider bars |
| MaxAdverse (MAE) | max(High) during hold (short: higher=worse) | data_provider bars |
| ScoreAtEntry | preserved from decision_log | decision_reader |
| MetricsAtEntry | JSON: MxV, RunUp, ATRX, RSI, REL_VOL, ScanChange, ConfidenceScore | decision_log |

### Auto-lesson rules (7 rules + special case)

1. LOSS + ATRX>3 → "High ATRX — consider archetype-specific limits"
2. LOSS + RSI>90 → "RSI 90+ at entry — momentum may have continued"
3. **LIVE_PAPER only** + duration<1h → "Very fast outcome — entry timing optimization"
4. WIN + MAE>SL → "MAE exceeded SL — stop-loss prevented larger loss"
5. MFE drop > 1.5× TP → "Stock fell X% (TP=10%) — trailing TP candidate"
6. EOD_CLOSE + profit → "EOD forced close with profit — consider extending hold"
7. EOD_CLOSE + loss → "EOD forced close with loss — earlier exit signal needed"
- Special: no bar data available → "MFE/MAE not computed"

Rule 3 excludes DRY_RUN to avoid false positives (DRY_RUN has artificial
exit timing within same minute as entry).

### Score historical accuracy

ScoreAtEntry preserves the score AT THAT TIME using AGENT_SCORE_VERSION
constant from config. When M7 changes the score formula:
1. Bump AGENT_SCORE_VERSION (e.g., "v2.6" → "v2.7")
2. New postmortems get tagged with new version
3. Old postmortems retain their original ScoreVersion
4. M7 Analytics can filter by version for apples-to-apples comparison

### Wire-in (position_manager hook)

`PositionManager.__init__` accepts optional `postmortem_engine` parameter.
After `_close_position()` updates the Sheet, if engine is provided:
1. Merges entry data (pos) + exit data (updates) into pos_with_exit
2. Calls engine.generate(pos_with_exit)
3. Wraps in try/except — postmortem failure doesn't fail the close

This makes M6 fully optional. Existing M5 tests pass with `postmortem_engine=None`.

- Critical: test_writes_17_column_row enforces schema integrity

## §A15: Data Sentinel — Agent #2 (Gatekeeper)

**Goal:** A validation layer that runs before The Trader, catching bad signals
and broken system state before they corrupt trading decisions. Born from a week
of silent-failure bugs (stale prices, 429 quota, None prices).

**Mental model:** Data Sentinel is a security guard at the entrance. Every signal
from FINVIZ passes through Sentinel's checks before The Trader evaluates it.
If something is wrong, Sentinel can BLOCK the signal (in active mode) or just
log it (in shadow mode).

**Two agents in production:**
1. The Trader — `agent/orchestrator.py` + `agent/trader/` — makes ENTER/SKIP decisions
2. Data Sentinel — `agent/sentinel/` — gatekeeper, runs before Trader

**Files:**
- `agent/sentinel/data_sentinel.py` — `DataSentinel` class, `SentinelResult` dataclass, `_log_sentinel_event()`, `get_sentinel()` singleton.
- `agent/sentinel/checks/` — 7 check modules.

**The 7 checks:**

Per-signal (4) — run on every signal:
- `completeness.py` — 7 required metrics present (ticker, score, price, mxv, run_up, atrx, rsi)
- `scan_freshness.py` — scan age: WARN at 3 min, BLOCK at 5 min
- `price_sanity.py` — price within [$0.01, $10,000] + OHLC consistency (low ≤ price ≤ high)
- `price_freshness.py` — FINVIZ price vs Alpaca live, BLOCK if delta > 2%. 60s in-memory cache.

System-level (3) — run once per orchestrator run:
- `position_sync.py` — if today_enters > 0 but open_positions = 0 → HALT (paper_portfolio failed to load)
- `quota_health.py` — Sheets writes/min: WARN at 50, BLOCK at 60
- `provider_heartbeat.py` — AAPL canary fetch, 3 consecutive failures within 5 min → HALT

**Modes (config.SENTINEL_MODE):**
- `shadow` — checks run, decisions logged to system_events, but NEVER block (DEFAULT)
- `active` — BLOCK actually prevents trader.evaluate()
- `off` — Sentinel disabled entirely

**Config flags (config.py):**
- `DATA_SENTINEL_ENABLED` (bool), `SENTINEL_MODE` (shadow/active/off)
- 7 per-check toggles: `SENTINEL_CHECK_*`
- Thresholds: `SENTINEL_PRICE_DELTA_MAX_PCT=2.0`, `SENTINEL_SCAN_MAX_AGE_MINUTES=3`,
  `SENTINEL_SCAN_MAX_AGE_BLOCK_MINUTES=5`, `SENTINEL_QUOTA_DEFENSIVE_THRESHOLD=50`,
  `SENTINEL_QUOTA_HALT_THRESHOLD=60`, `SENTINEL_PRICE_MIN_USD=0.01`, `SENTINEL_PRICE_MAX_USD=10000.0`

**Observability:**
- Events written to `system_events` sheet (schema: Timestamp, EventType, Severity, Component, Message, Details, ActionTaken). Only BLOCK/WARN written (never ALLOW — quota).
- Dashboard page "🛡️ Sentinel Events" — KPIs, events table, breakdown by check/reason.
- Health check S1 (`check_24_sentinel_health`) — verifies 9 files, config flags, system_events reachability.

**Integration in orchestrator.py:**
- `sentinel.check_system()` runs once after build_account_state — HALT if BLOCK
- `sentinel.check_signal()` runs per signal before `trader.evaluate()` — skip signal if BLOCK
- Counter `sentinel_blocks` in run summary

**Activation path (shadow → active):**
- Currently SHADOW mode since 2026-05-11.
- Decision criterion: after several days of clean shadow logs (low false-positive
  BLOCK rate, ideally < 5% of signals), switch SENTINEL_MODE='active'.
- Active mode means bad signals are actually blocked before reaching The Trader.

**Iron Rule additions:**
- Sentinel checks MUST never raise — exceptions caught, treated as WARN, never block the loop
- Sentinel writes MUST be graceful — system_events write failure logged, never breaks trading
- Only BLOCK/WARN written to system_events — ALLOW never written (quota protection)

**Status:** Built 2026-05-11 (Phase 1-3). Observability completed 2026-05-16
(system_events logging + dashboard page + health check). Running in SHADOW mode.

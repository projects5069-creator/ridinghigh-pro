# RidingHigh Pro — Session Protocol
*הפרוטוקול החי לפתיחת יום וסגירת יום.*
*מקור האמת היחיד לטקסים האלה. אם הזיכרון או הסקיל מתנגשים — הקובץ הזה קובע.*

---

## 0. עיקרון העל

כל העבודה נשמרת בקבצים חיים בריפו. הזיכרון של Claude (chat) מחזיק רק
**מצביעים וחוקים** — לא מצב. בכל פתיחת יום אני בונה את התמונה מאפס
מהקבצים, לעולם לא מהזיכרון. בכל סגירת יום כל בקשה חדשה הופכת ל-TASK
רשמי ב-Backlog לפני הסגירה, ונכתב SESSION_HANDOFF.

**מצביע, לא משכפל:** ההודעות וה-handoff מצביעים על מקורות חיים
(PK, MASTER, Backlog) — הם לא משכפלים עובדות לתוכם, כי עובדות משוכפלות
מתיישנות. דוגמה: לא לכתוב גרסת PK קשיחה אלא "ה-PK העדכני" + לקרוא אותו.

---

## 1. זיהוי כוונה — לא מילת קוד (Intent, not keyword)

עמיחי **לא** משתמש במילת-קוד קבועה. בכל יום הוא מנסח אחרת. תפקידי לזהות
את **הכוונה** מההקשר ולהפעיל את הטקס הנכון — בלי להתקבע על ביטוי מסוים.

**כוונת פתיחת יום** — כל ניסוח שמשמעו "מתחילים לעבוד / איפה עומדים / מה
על הפרק". דוגמאות לא ממצות: "בוקר טוב מתחילים", "מה המצב היום", "קדימה
ליום חדש", "איפה עצרנו", "תן סטטוס", "יאללה עבודה", או הדבקת פלט recon
בבוקר.

**כוונת סגירת יום** — כל ניסוח שמשמעו "מסכמים / עוטפים / שומרים לפני
שהולכים". דוגמאות לא ממצות: "סוגרים", "מספיק להיום", "תעטוף את היום",
"סיימנו", "תשמור מה שעשינו", "לקראת סיום".

**בספק — לשאול, לא להניח.** אם לא ברור אם זו כוונת טקס או שאלה רגילה,
שאל קצר ("להפעיל פתיחת יום מלאה?") לפני הפעלת טקס שלם.

## 2. פתיחת יום (Session Open)

כשמזוהה כוונת פתיחת יום, בצע לפי הסדר:

### שלב 1 — סקיל + אימות תקינות
1. קרא ~/.claude/skills/rhpro-live/SKILL.md והצג שורת סקילים עם line count אמיתי.
2. **הרץ `bash ~/RidingHighPro/scripts/check_skills_integrity.sh`** — חובה. אם FAIL, עצור ותקן לפני שממשיכים. (מונע את באג 26/5 שבו ה-superpowers שונו ל-.bak והסקילים נעלמו בשקט.)

### שלב 2 — פרומפט recon אחד ל-Claude Code
פרומפט יחיד, עטוף ב-.rh-run.sh, פותח ב-wc -l של הסקיל המקומי, שקורא
מהקבצים בלבד (דינמי — בלי תאריכים קשיחים):
1. PK head — version + changelog (head -80 על docs/RidingHigh_Pro_PK_v2.md)
2. ה-SESSION_HANDOFF העדכני ביותר — ls -1t docs/SESSION_HANDOFF_*.md | head -1 ואז cat עליו. לעולם לא תאריך קשיח.
3. PROJECT_STATE.md (בשורש)
4. git log --oneline -10
5. MASTER_TASK_LIST מלא + backlog task list --plain מלא
6. בדיקות חיות: decision_log היום, ENTERs, Filter 9, reconciliation, agent_minute

### שלב 3 — קריאה לפני דיבור
אחרי הפלט: קרא קודם את ה-PK ואת ה-handoff העדכני, ורק אז דבר. אל תתחיל מהזיכרון.

### שלב 4 — הודעת פתיחה מלאה
הרכב הודעה הכוללת:
- מצב מערכת — שעה (Peru), PK עדכני, commit אחרון, סוכנים פעילים
- כל המשימות ללא חיתוך — MASTER (כל השלבים) + Backlog (כולן) + חדשות
- ספירה מול הרשימה החיה — הצג מספר משימות מול המספר ברשימה החיה
- חוסמים אסטרטגיים
- כפילויות / אי-התאמות לטיפול ניהולי
- תעדוף מוצע ליום (חובה / חשוב / רצוי)

## 3. סגירת יום (Session Close)

כשמזוהה כוונת סגירת יום, בצע לפי הסדר:

### שלב 1 — סריקת בקשות היום
עבור על הסשן וזהה כל בקשה/משימה חדשה שעלתה — כולל בקשות בעל-פה. כל אחת
נכנסת ל-Backlog מיד כ-TASK (backlog task create). בסגירה נדון אם רלוונטית.

### שלב 2 — כתיבת SESSION_HANDOFF חדש
פרומפט ל-Claude Code שכותב docs/SESSION_HANDOFF_<תאריך-היום>.md בפורמט
הקבוע (ראה §4). התאריך נקבע דינמית מ-TZ=America/Lima date.

### שלב 3 — עדכון PK (חובה — ללא יוצא מן הכלל)
עדכן PK בכל אחד מאלה: סגירת יום / כתיבת SESSION_HANDOFF / כל בקשה
לעדכון משימות. תמיד קרא את הגרסה החיה (grep Document version) ואז bump
לגרסה הבאה + שורת changelog על מה שנעשה. גם אם רק תיעוד/תשתית — מעדכנים.
אם נגעת בקוד מהותי (נוסחאות / workflows / sheets / config / health
checks) — זה חוזה Anti-Drift, חובה כפולה.

### שלב 4 — עדכון סטטוס משימות
סמן ב-Backlog: הושלמו -> Done, בוטלו -> Cancelled/archive (לא להשאיר תקוע).

### שלב 5 — אימות חי
ודא: git status נקי, קובץ ה-handoff קיים, Backlog מעודכן.

### שלב 6 — שאלת ביקורת לפני סגירה
שאל את עצמך במפורש: "האם יש בקשה מהיום שלא הפכה ל-TASK?" אם כן —
היא נכנסת לפני שסוגרים. הטקס לא נסגר עד שהתשובה היא "לא".

## 4. פורמט SESSION_HANDOFF (התבנית הקבועה)

כותרת (תאריך+יום) / משך / commits / משימות שנסגרו / עקרון תיעוד /
קריאה חובה לסשן הבא / טבלת commits / בעיות פתוחות מדורגות + תיוג
מאומת-מול-השערה / משימות פתוחות (מספרים מול הרשימה החיה) / תעדוף למחר /
לטיפול ניהולי / חתימה.

הפורמט המדויק נלקח מה-handoff העדכני ביותר בריפו
(ls -1t docs/SESSION_HANDOFF_*.md | head -1) — לא מקובץ עם תאריך קשיח.

## 5. חוקי ברזל החלים על שני הטקסים

- חקירה לפני כתיבה — קרא קוד/סכמה/נתונים אמיתיים לפני כל מסקנה.
- Claude Code בלבד — כל הוראה = בלוק inline אחד, עטוף .rh-run.sh.
- עברית, לשון זכר, בלי ask_user_input_v0.
- בלי לערבב מטרות — טקס פתיחה/סגירה לא מערבב בתוכו חקירת מערכת לא
  קשורה. אם עולה אנומליה תוך כדי — רושמים אותה כ-TASK, לא רודפים אחריה
  באמצע הטקס.
